# Ultra-Goal

[English](./README.md) | [中文](./README_cn.md)

> 为 Codex、Claude Code 等 Agent 设计的多智能体编排 Skill，用于在无人值守状态下持续推进长周期、大规模工程任务。 

## 安装

直接让 Agent 安装 Ultra-Goal skill：

> Install this skill: <https://github.com/Azure99/ultra-goal/tree/main/skills/ultra-goal>

也可以通过 `skills` CLI 安装：

```bash
npx skills add Azure99/ultra-goal
```

## 设计背景

过去一年中，Superpowers、GSD、OpenSpec 等 spec-driven 工作流已经被广泛接受；“先明确规约与计划，再交由 Agent 实现”的 SDD（Spec-Driven Development）理念，也几乎成为 AI 辅助开发中的事实标准。

然而，在实际执行 SDD 流程时，仍然存在三个难以回避的痛点：

1. **决策成本仍由使用者承担。**
   在编写 spec / plan 的阶段，使用者依然需要在多个候选方案之间做出选择，例如数据结构、API 风格、迁移路径等。每一个关键分叉点，最终仍需要人工裁定。

2. **SDD 天然不适合超长程任务。**
   对于需要持续数小时甚至数天的大型任务，在真正开始实现之前，几乎不可能预先穷尽所有细节。spec 越长，越容易偏离现实约束；偏差累积得越早，后续实现与修正的成本就越高。

3. **动态调整成本较高。**
   一旦实施到中途，发现最初的 spec / plan 不可行——例如选错了库，或某个模块的复杂度被严重低估——常规 SDD 工具往往难以顺畅地回退、修正或重新规划。使用者通常需要中断会话、重新补充背景，甚至推倒重来。

## 设计理念

Ultra-Goal 将工作组织为连续推进的阶段循环。每个阶段都会独立完成规划、实现、验收和提交；阶段结束后，系统会回顾完整 roadmap，并根据已交付成果，动态地新增、删除、合并、拆分或重排后续阶段。

每个阶段内部固定包含四个步骤：

1. **规划（Planning）**
   启动 2 个子 Agent 独立探索，分别产出初版计划，由主控 Agent 作为最终裁决者进行仲裁，形成最终实施计划。

2. **实现（Implementation）**
   启动 1 个子 Agent 按照实施计划执行，并完成必要的自测。

3. **验收（Acceptance）**
   启动 2–3 个子 Agent 独立审查，进行交叉补充与反驳，最后由主控 Agent 综合判断该阶段是否通过验收。

4. **提交（Commit）**
   若通过验收，则提交一个 git commit；若未通过，则退回实现 Agent 返工。

这一设计与传统 SDD 工具存在几个本质区别：

* **决策不再依赖人工介入。**
  所有 plan 中的 trade-off，以及 review 阶段产生的争议，均由多个子 Agent 评估、投票，并由主控 Agent 仲裁，不再交还给使用者裁定。

* **计划是持续演化的。**
  roadmap 会在每个阶段完成后被回顾和修订。所有变更都会记录在 `roadmap-changelog.md` 中。

* **目标可以保持适度抽象。**
  使用者只需要提出一个相对清晰的目标，无需在任务开始前预先设计完整的实施细节。

## 适用场景

**Ultra-Goal 不适合小型任务。**

启动一个 Ultra-Goal 流程，意味着会投入大量 token 用于规划、评审、投票和仲裁。对于“添加一个按钮”或“修复一个 typo”这类局部改动，这种开销通常并不划算。

Ultra-Goal 适合的场景包括：

* 需要持续数小时乃至数天的大型工程任务，例如大型重构、跨技术栈迁移、从零搭建完整模块；
* 使用者无法持续盯着屏幕，希望任务在无人值守状态下持续推进；
* 需要并行推进多个项目，希望降低持续决策与监督成本。

## 模型选择

* **主控模型应尽可能选择能力强的模型。**
  在 `Codex + gpt-5.5 xhigh` 组合下表现良好。主控 Agent 负责总体调度，如果模型能力不足，整个流程很容易失控。

* **子 Agent 默认沿用主控模型。**
  如果对成本敏感，可以显式指定子 Agent 使用更小的模型，例如 `gpt-5.4`，甚至 `gpt-5.4 mini`。

## 使用建议

### 1. 目标应清晰，但不应包含过多实施细节

一个合适的目标示例：

> “把这个 Express 项目迁移到 Fastify，保持现有 API 行为不变，所有现有的集成测试和 E2E 测试必须继续通过。”

过于模糊的目标不适合：

> “改一下项目。”

但目标也不需要写成具体实施步骤：

> “把 Express 迁移到 Fastify，第一步先改 `server.ts`，把 `app.use` 都换成……”

这类实现细节会挤占子 Agent 的探索和规划空间，反而可能降低最终质量。

### 2. 尽量提供端到端的验收方法

仅依赖单元测试通常是不够的。子 Agent 完全可能让测试全部通过，但实际功能已经无法正常运行。

建议在目标中明确端到端验收手段，例如：

* **Web 应用**：让验收 Agent 使用 `playwright-cli` 控制真实浏览器，对关键用户路径进行交互验收和视觉验收。
* **CLI 工具**：执行真实命令，校验 stdout、退出码和生成文件。
* **服务端 API**：启动真实服务进程，使用 `curl` 或 HTTP client 验证关键 endpoint 的行为。

### 3. 需要显式调用

Ultra-Goal 的 `SKILL.md` 设置了 `disable-model-invocation: true`，因此模型不会自动套用这一流程。使用者必须显式发起调用。

这是有意为之的设计：Ultra-Goal 的运行成本较高，不适合被自动触发。
